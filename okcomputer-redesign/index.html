A comprehensive redesign of the okCOMPUTER desktop experience. We audited 183 issues across the current app and are restructuring the entire surface architecture, user journey, and onboarding flow.

What changed and why: okCOMPUTER v0.6.4 has four separate UI surfaces — HUD overlay, tray popover, main window, and notification toasts — that feel like four different apps. Users don't know where to look. The same information (voice results, TTS controls, recording state) appears in multiple surfaces simultaneously. The onboarding teaches button mechanics instead of product value, and there's no coherent journey from sign-up through progressive feature discovery.

The redesign: We're collapsing to two surfaces. The HUD and panel become a single morphing window — pill → card → full panel — that serves as the always-present entry point ("the dock"). The notification window is eliminated entirely. A new RPG-style "Build Your Chief of Staff" onboarding replaces the mechanical permissions wizard. Features unlock progressively through usage, not time.
Problem
4 fragmented surfaces with overlapping information. 183 documented issues (7 critical, 38 high). Voice results displayed in both HUD and notification simultaneously. TTS controls duplicated. 5-tab popover where Home duplicates all other tabs. Onboarding forces users through 3 dictation modes before proceeding — never mentions meeting recording, the primary use case.
Solution
2 surfaces with clear roles. The HUD morphs into the panel as one continuous window — clicking or hovering expands it. All notification types are absorbed. Onboarding becomes an RPG "build your agent" experience with a conversational interview. Features progressively unlock: Voice → Meetings → Agency → Proactive.
New Architecture
HUD/Panel — "The Dock." Always present on screen. A small pill that morphs into a full panel. Primary interface for real-time activity, controls, settings, and onboarding.

Main Window — "The Workspace." Loads okwow.ai for deep content: full transcripts, chat threads, vault, insights. Opens only when the user explicitly navigates there from the panel.

HUD: The Dock

The HUD is a small floating element anchored to the bottom of the screen. It's always visible while the user works in other apps. It has 5 states that it transitions between based on what's happening — from a barely-visible dormant bar to a fully expanded content panel. The entire element is one continuous shape that morphs between states (pill → card → panel). It's hidden from screen share via macOS content protection.

Activation Feedback (Latency Mitigation)

There is perceivable latency between pressing Fn and the microphone actually activating. This "dead air" makes users unsure if it worked. We bridge the gap with staged sensory feedback — animation first, then sound when the mic is truly ready.

1. Fn Press
Activating...
Instant animation. HUD morphs to active state immediately. Bars appear flat/static. No sound yet — just visual acknowledgment.
2. Mic Ready
Listening...
Sound + live bars. Activation chime plays. Bars start responding to actual mic input. User knows: "I can speak now."
3. Fn Release
Processing...
Animated spinner. No sound. Visual spinner communicates "I'm working on what you said." Transitions to expanded result card.

Hover tooltip: When hovering over the dormant HUD pill, show "Click or hold fn to start dictating." Teaches the interaction model progressively — users learn by discovering, not by being told in onboarding.

Rollover Tooltips

Tooltips teach the interaction model progressively. Users discover shortcuts by hovering, not by reading onboarding screens. Tooltips should feel like hints, not instructions — brief, contextual, and only shown when the user is already engaged with the element.

Panel States

When the user clicks the HUD pill or the tray icon, the HUD morphs into its full panel view. This is the same window expanding — not a separate popover. The panel content changes based on context: default shows upcoming meetings and recent activity, after a voice command it shows the conversation, and when a meeting is detected it shows an attention card. Quick actions (Record, Note, Chat) are always pinned to the bottom.

The current 5-tab popover (Home, Meetings, Notes, Voice, Computer) is replaced by this single scrollable view with contextual sections that appear and disappear based on state. Every item in the panel is a link to the main app — meetings open the meeting detail, notes open the vault, commands open the conversation thread.

Recording State

When a meeting recording starts, the panel transforms entirely into a focused recording view. The default sections (upcoming, recent, quick actions) are replaced by two full-width tabs: "Meeting" for the live transcript and scratchpad, and "Insights" for proactive vault suggestions and detected action items. A purple dot appears on the Insights tab when new insights arrive.

The panel is hidden from screen share participants via macOS content protection. A "Hidden from screen share" indicator is shown at the bottom. Pause and Stop buttons are equal-sized at the bottom of both tabs. The button is Pause (stops all transcription), not Mute — muting is platform-specific and doesn't control the Recall SDK audio capture. The panel can be minimized via the (—) button; when minimized, the HUD pill shows "Recording · 12:34" and clicking it reopens the panel. An upcoming meeting alert appears at the top when the next meeting is within 5 minutes.

Onboarding: Build Your Chief of Staff

The current onboarding is a 3-step permissions wizard that teaches mechanics ("hold Fn to dictate") but never shows value. Users are forced to try all 3 dictation modes before proceeding. Meeting recording — the primary use case — isn't mentioned. After completing onboarding, the user lands on an empty Home tab with no guidance.

The new onboarding is an RPG-style "build your character" experience. The user creates their AI chief of staff — choosing name, avatar, communication preference (voice or text), their role, and work style. Permissions are requested in parallel as "agent setup" with human explanations ("Microphone — so you can record meetings and take voice notes"). While permissions grant, keyboard shortcuts are taught as they become available.

After creation, the agent conducts a conversational interview following the Indi v7.0 principles: it researches the user first, opens with genuine curiosity (not "how can I help?"), and follows the user's thread through 3-5 turns. This is skippable — context builds gradually through usage instead. The interview is available via voice (hold Fn) or text. After the interview, the agent guides the user to a first real action — not "you're set" but "let's build/explore/think through something together." The main app window never opens during onboarding; that's earned when the user clicks something that needs the full workspace.

Progressive Disclosure

okCOMPUTER has a deep feature set — recording, transcription, AI summaries, dictation, voice commands, computer use, permission tiers, vocabulary, shortcuts, style rules. Showing all of this on first launch is overwhelming. Instead, features unlock based on usage, not time. Tiers are implicit — the UI never announces "you unlocked tier 3." The agent introduces capabilities when they become relevant. A "Skip — Unlock Everything" bypass is always available for power users.
